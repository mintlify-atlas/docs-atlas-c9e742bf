---
title: "Firebase Cloud Storage"
description: How MundeApp uploads and serves destination hero images via Firebase Cloud Storage
---

Each destination's hero image is stored in Firebase Cloud Storage. The image is uploaded when a destination is created or edited with a new photo. The resulting download URL is saved in the Destination's Firestore document and used to display the image throughout the app.

## How images are stored

Images are uploaded by `FirebaseService` when `saveDestination()` or `updateDestinations()` is called. The upload flow is:

1. The user selects an image from their device gallery in `FormDestinationActivity`.
2. The local image `Uri` is passed to `FirebaseService`.
3. `FirebaseService` uploads the image to Cloud Storage.
4. After the upload completes, Firebase Storage returns a permanent download URL.
5. That URL is written to the `imageUrl` field in the Firestore document.

Once stored, the `imageUrl` is used by [Glide](https://github.com/bumptech/glide) to load the image in the `RecyclerView` cards on the main screen and in the image preview on the form screen.

<Note>
  When a destination is edited without selecting a new image, `updateDestinationWithoutImage()` is called instead. This updates all other fields in Firestore and leaves `imageUrl` unchanged, so the existing Storage file continues to be used.
</Note>

## Required permissions

The app declares the following permissions in `AndroidManifest.xml` to read images from the device gallery:

```xml
<uses-permission android:name="android.permission.INTERNET"/>
<uses-permission android:name="android.permission.READ_MEDIA_IMAGES" />
<uses-permission android:name="android.permission.READ_EXTERNAL_STORAGE"
    android:maxSdkVersion="32" />
```

| Permission | API level | Purpose |
|---|---|---|
| `INTERNET` | All | Required for all network calls including Firebase Storage uploads and downloads |
| `READ_MEDIA_IMAGES` | 33+ (Android 13+) | Read images from the device gallery |
| `READ_EXTERNAL_STORAGE` | Up to 32 (Android 12 and below) | Read images from the device gallery on older API levels |

<Note>
  `READ_MEDIA_IMAGES` and `READ_EXTERNAL_STORAGE` are for reading local gallery images — they are not required for Firebase Storage itself. Firebase Storage operations use the `INTERNET` permission.
</Note>

## Security rules

Firebase Storage security rules control who can upload and download files. The default rules for development allow any authenticated user to read and write:

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    match /{allPaths=**} {
      allow read, write: if request.auth != null;
    }
  }
}
```

Apply these rules in **Firebase Console → Storage → Rules**.

<Warning>
  The default rules generated by Firebase in test mode may allow unauthenticated access. Replace them with the rules above before going to production.
</Warning>

### Applying Storage rules

<Steps>
  <Step title="Open Storage Rules">
    In the Firebase Console, go to **Build → Storage** and click the **Rules** tab.
  </Step>
  <Step title="Replace the existing rules">
    Paste the rules above into the editor.
  </Step>
  <Step title="Publish">
    Click **Publish**. The rules take effect immediately.
  </Step>
</Steps>

## Known limitation — orphaned images

When a destination is deleted, `FirebaseService.deleteDestinations(id)` removes the Firestore document for that destination. However, the associated image file in Cloud Storage is **not** automatically deleted in the current implementation.

Over time, Storage can accumulate image files that are no longer referenced by any Firestore document. These orphaned images continue to count toward your Storage usage.

<Tip>
  You can manually delete orphaned images from **Firebase Console → Storage** by browsing the bucket and removing unused files. For an automated solution, implement a Firebase Cloud Function triggered on Firestore document deletion that reads the `imageUrl` field and deletes the corresponding Storage object before the document is removed.
</Tip>
